# SaaS Point 6 - Security Monitoring, Alerting, and Evidence

## 1. Objective

This document provides the current policies, procedures, and technical controls used to monitor attacks, malware, and anomalous behavior across the network, infrastructure, and application layers for the IGS Billing project.

This point should be interpreted together with [01-assessment-context.md](../01-assessment-context.md).

## 2. Current Compliance Position

**Status:** Partially compliant.

Repository evidence confirms that the project implements application-level monitoring, traceability, observability, and operational verification controls. However, the current technical and security documentation also confirms that centralized SIEM log aggregation and alert correlation remain open remediation items. Accordingly, full compliance cannot yet be asserted, and real SIEM alert samples cannot be produced from the repository alone.

## 3. Scope of Verifiable Evidence

The following controls are verifiable from the current repository state:

- Application monitoring and observability.
- Authenticated health checks for critical components.
- Security logging and auditing for sensitive models.
- Logging channel configuration and local retention settings.
- Existing security, hardening, and assessment documentation.

The following items are outside the verifiable scope of the repository because they depend on externally managed production operations:

- A live SIEM platform in operation.
- Centralized event correlation dashboards.
- Real alerts generated by infrastructure, WAF, EDR, NDR, or network controls.
- SOC or monitoring-desk operating procedures maintained outside the codebase.

## 4. Identified Controls and Procedures

### 4.1 Availability and Critical Component Monitoring

The platform exposes authenticated health check endpoints to validate database, cache, storage, route, and application status. Existing documentation recommends periodic invocation of these endpoints and alerting when any component returns an `unhealthy` status.

Verified controls:

- General and component-specific health checks.
- Documented automated monitoring recommendation at 1 to 5 minute intervals.
- Planned integration path for tools such as Nagios, Zabbix, Prometheus, DataDog, and New Relic.

### 4.2 Security Logging and Local Retention

The logging configuration includes a dedicated security channel with daily rotation and 30-day local retention. The codebase also supports optional Slack and Papertrail routing, but the repository does not provide sufficient operational evidence to conclude that these integrations are active in production.

Verified controls:

- `security` log channel using the `daily` driver.
- Log file output at `storage/logs/security.log`.
- 30-day configured retention period.
- Environment-driven capability to forward events to Slack or Papertrail.

### 4.3 Change Auditing and Traceability

The project uses OwenIt Laravel Auditing for sensitive entities and exposes audit retrieval endpoints across multiple functional domains. Audit configuration captures the authenticated user, IP address, user agent, and request URL associated with each event.

Verified controls:

- Auditing enabled by default.
- Audited lifecycle events: `created`, `updated`, `deleted`, and `restored`.
- User resolution compatible with `web`, `api`, and `sanctum`.
- IP, User-Agent, and URL resolvers.
- Documented evidence of at least 27 audited models.

### 4.4 Application Observability

The production stack includes Laravel Nightwatch for observability and Laravel Telescope for development diagnostics. The hardening guide explicitly requires Telescope to be disabled in production and Nightwatch to be enabled in production.

Laravel Nightwatch improves application-level visibility, performance monitoring, telemetry, and error tracking. However, based on the current project documentation and available evidence, Nightwatch must be treated as an observability control, not as a substitute for a centralized SIEM platform capable of multi-layer security event correlation.

Verified controls:

- `laravel/nightwatch` dependency included.
- `laravel/telescope` included as a development dependency.
- Hardening checklist requiring `TELESCOPE_ENABLED=false` in production.
- Hardening checklist requiring `NIGHTWATCH_ENABLED=true` in production.

### 4.5 Monitoring Position in the Security Assessment

The current security assessment classifies the logging and monitoring domain as **partial** because centralized SIEM capabilities are still absent. The same assessment leaves SIEM centralization and alerting as open remediation items.

Verified conclusions:

- The project has strong technical controls at the application layer.
- The explicit gap is the absence of centralized SIEM capabilities.
- The documented recommendation is to integrate ELK, Datadog, or CloudWatch.

## 6. Recommended Requirement Wording

IGS Billing currently implements monitoring and traceability controls at the application layer, including authenticated health checks for critical components, a dedicated security logging channel, auditing over sensitive models, and production observability through Laravel Nightwatch. These controls support operational issue detection, change traceability, and application-level visibility.

Laravel Nightwatch contributes to production observability, telemetry, and error tracking; however, it does not, by itself, satisfy the role of a centralized SIEM platform for cross-layer security event correlation. Based on the current security assessment, this requirement remains partially compliant because centralized log aggregation, SIEM correlation, and real SIEM alert evidence remain open items.

## 7. Open Items Required for Full Compliance

To move this requirement from partial to full compliance, additional operational evidence external to the repository is required:

- Screenshots or exports of real SIEM alerts.
- Evidence that application, infrastructure, network, and WAF logs are centrally ingested.
- An approved monitoring and incident escalation procedure.
- Evidence of periodic alert review and response handling.

## 9. Conclusion

Based on the evidence currently verifiable in the development repository, this point should be answered as **partially compliant**, supported by real logging, auditing, health check, and application observability controls. The missing evidence required for full compliance is the effective operation of a centralized SIEM platform and the attachment of real alerts generated by that platform.
